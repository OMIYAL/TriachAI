# 🚀 TriArchV5 — v2026.6.26.2131 (27 June 2026)

## ✨ New

**Quick Report — report issues from inside the app**
A new **Quick Report** button in the application toolbar lets you capture a screenshot and submit a problem report without leaving the app. Feedback lands directly in the team's issue tracker — no switching to email or a separate tool needed.

**Document Editor in the Documents section**
You can now open and edit documents directly inside the **Documents** area. No more downloading a file, editing it externally, and re-uploading — the entire edit cycle happens in one place.

**Packaging Activity — build your final package without leaving the app**
A new **Packaging** workflow step lets coordinators assemble a finalised submission package from within the platform. Workflow Outputs and Applicant Submittals appear side by side in two independently-scrollable panels. A rich document viewer lets you arrange, review, and finalise pages across a clear three-step flow (Select → Edit → Finalize). Conditional reviewer notes are automatically woven into the merged document.

**Invoicing: Storefront applicants now included, plus KPI summary and improved filtering**
The Invoicing screen now shows a complete picture:
- **Storefront applicants** — invoices for requests submitted via the Storefront portal are now included alongside staff-created ones.
- **KPI summary** — headline totals (outstanding, paid, overdue) appear at the top of the Invoicing page.
- **Better filters** — new and improved filter controls make it faster to find exactly the invoices you need.

**Conditional reviewer notes automatically included in the Final Package**
Conditional notes that reviewers add during their review step are now automatically embedded in the final merged document. The right content appears in the output without any manual copy-paste.

**Reject handling in reviewer blocking steps**
When a reviewer submits a **Reject** decision at a blocking workflow step, the case now correctly follows the rejection path. Previously, rejections at these steps could stall the workflow or route the case incorrectly.

## 💪 Improvements

**Reviewers see only their assigned requests in the Service Request list**
Staff members without full management access now see only the service requests assigned to them, rather than the full request list. This makes the reviewer queue cleaner and easier to work from.

**Service Request list: date filters now clear correctly**
Clicking "Clear" on the Submitted Date filter now properly resets the date picker. Previously the picker retained its displayed value even after the filter had been cleared.

**Service Request list page enhancements**
Additional layout and usability improvements to the Service Request list make it easier to find, sort, and take action on requests.

**Packaging work surface redesigned to match Document Review**
The Packaging area has been refreshed with a consistent Triarch visual style and the same toolbar conventions as Document Review. The file tray is split into **Workflow Outputs** (top) and **Applicant Submittals** (bottom). Navigation uses **Save & Next** and **Next** buttons, keeping the primary action in one place across all three steps. Merge failures now surface the actual error message rather than a generic "Merge failed" notice.

**Output documents now labeled by their review step**
When an applicant downloads output files (review reports, letters, certificates), each file now shows the **name of the activity that produced it** — for example, "Plan Review Report" — so it is clear what each document is without having to open it.

**Only completed outputs are shown to applicants**
Applicants now only see documents from **fully decided review steps**. Reports from steps still in progress are no longer exposed before a decision has been reached.

**Correction cycles now labeled R1, R2, R3 (Revision)**
The correction history drawer and document version badges now use **Revision** numbering (R1, R2 …) instead of Version (V1, V2 …). The revision badge is also displayed in a more prominent red colour to draw attention to cases currently under revision.

**Upload limit raised from 25 MB to 100 MB**
Files up to **100 MB** can now be uploaded across all upload points: document uploads in forms, supporting documents on the Storefront application, and attachments in the document editor.

**Fee Schedule name now shown in the step editor**
When editing a workflow step that already has a Fee Schedule linked, the picker now correctly shows the **name of the assigned schedule** instead of displaying blank. Coordinators can now confirm which fee schedule is attached without reopening the picker.

**"Mark All Sections Reviewed" button (renamed from "Clear all reviewed")**
The checklist button that marks every section as reviewed has been renamed to **"Mark All Sections Reviewed"** to describe accurately what it does. The old label "Clear all reviewed" was misleading.

**ControlRoom: page help guides and table refresh button**
ControlRoom pages now include contextual **page help** and a one-click **table refresh** button to reload data without a full page reload. The "Assign to All" activity panel bug has also been fixed, and Submission Guidance documents now automatically use the uploaded file's name as the document title.

**Pre-configured roles, forms, checklists, and fee schedules on new environments**
New environments now ship with a fuller set of default configuration — roles, form templates, checklists, and fee schedule definitions — reducing the time from first login to a working setup.

## 🐞 Fixes

**SLA due dates are now accurate**
SLA timers now start when a workflow step is **first activated**, not when the service request was originally submitted. The timer also **pauses while a request is on hold** — for example, while waiting for a fee payment or a correction to be submitted — and resumes when the hold is lifted, extending the due date by exactly the hold duration. The Operations Dashboard now reflects these corrected timers: untracked steps are excluded from adherence figures, and requests that have resumed from a hold are no longer counted twice.

**Storefront applicants can see their submitted requests again**
Applicants using the **Storefront portal** can once again view all their submissions in **My Requests**. A regression had caused the list to return empty. Additionally, the page shown after completing a payment no longer returns a 404 error.

**Service requests no longer close early in multi-step workflows**
A service request now only transitions to Closed when there are genuinely **no remaining active or pending steps**. Previously, if the "final step" marker had become stale — for example because a coordinator added steps to an already-running request — the case could close mid-workflow before all work was complete.

**Service Request list: Closed requests handled correctly**
An issue causing unexpected behaviour on the Service Request list for requests in the Closed status has been resolved.

**Invoicing stability fixes**
Multiple display and reliability issues in the Invoicing module — introduced alongside the new Invoicing features in this release — have been resolved.

**Three tracked change requests resolved (CR167, CR98, CR164)**
Three change requests logged by the team have been investigated, addressed, and closed.
